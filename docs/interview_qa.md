# 面试 QA 手册

> 覆盖基于此项目可能被问到的高频面试题，分为项目深挖、系统设计、技术原理、金融场景四个维度。每道题提供回答框架，而不是背诵答案。

---

## Part 1：项目深挖

### Q1：你为什么选 Skyvern 而不是 UiPath、影刀，或者自己用 Playwright 写脚本？

传统 RPA（无论哪个产品）的本质问题是依赖页面结构——靠 XPath 或 CSS 选择器定位元素，页面改版一次脚本就失效。在需要对接大量第三方系统的金融机构，这个问题尤其严重，因为他们没有能力控制外部网站（监管报送系统、银联、各银行网银）何时改 UI。

Skyvern 用 LLM + 截图来「理解」页面语义，即使页面改版，只要按钮的含义还在，Agent 就能继续工作。这种鲁棒性在金融场景里价值很大。但代价是执行成本高，每步都要调 LLM，所以它适合执行频次相对低、但灵活性要求高的场景。选它是因为目标业务场景符合这个特征。

---

### Q2：你说做了三维度权限模型，为什么不用简单的 org_id + role 就够了？

这道题最容易暴露权限设计是不是真的思考过。

单一 org_id 隔离的问题是：在同一家银行里，对公信贷部的操作员不该看到个人金融部的 RPA 任务；但风险管理部的员工需要看到全组织所有任务；合规审计部的人需要能审批任何业务线的高危操作。如果只按 org_id 隔离，要么所有人都能看全（合规问题），要么风控/合规的跨部门需求没法满足。

三维度模型——部门（行政归属）× 业务线（功能分类）× 角色（操作权限）——是覆盖这些真实场景的最小设计。其中有两个关键约束：一是 operator 和 approver 互斥，同一人不能同时持有（对应金融双重控制原则），这个约束放在数据库层面而不只是应用层，防止漏检；二是风控/合规部门的跨组织权限用专门的 special_permissions 表管理，不是通过角色上移来实现，避免角色膨胀。

---

### Q3：operator 和 approver 互斥，如果有人改数据库直接插入怎么办？

这道题考察防御性设计意识。

应用层的权限检查是第一层，但应用层可以绕过。所以这个约束放在了数据库层面——`user_department_roles` 表上有 CHECK 约束（或触发器），数据库层面拒绝同一个 user_id 在同一 department_id 下同时存在 operator 和 approver 记录。即使直接操作数据库也会被拒绝。

这是金融合规对双重控制要求的工程落地，不是随便加的约束。

---

### Q4：审批等待机制是怎么实现的，为什么用 Redis Pub/Sub？

任务执行到高危操作时，把审批请求写入数据库，然后订阅一个以 `approval_id` 为 key 的 Redis channel，挂起当前协程等待消息。审批员在前端点击「通过」时，服务端向该 channel publish 消息，协程解挂，根据消息内容继续执行或终止任务。

为什么不用数据库轮询：有延迟，且频繁查数据库有压力。为什么不用 Kafka：审批场景并发量不大，金融机构一天触发人工审批的 RPA 操作通常是个位数到几十条，引入消息队列是过度设计，Redis 已经在系统里用于 action 缓存，复用更合理。

### Q5：高危操作识别会有误报吗，误报率怎么控制？

会有，而且控制误报比控制漏报更难。

两阶段的设计本身就是为了平衡：第一阶段关键词+正则的召回率高但误报多（「确认」这个词在很多普通操作里也会出现），第二阶段 LLM 结合页面上下文精判，能过滤掉大部分误报。

但误报率还是无法归零。我的设计取舍是：在金融场景下，宁可多触发一次审批（误报），也不能漏掉一次真正的高危操作（漏报）。误报的代价是审批员多看了一次请求，漏报的代价是 AI 直接执行了一笔未授权的转账。这个不对称决定了策略偏向保守。

---

### Q6：LLM 失败了怎么处理，NEEDS_HUMAN 状态是什么？

LLM 失败分两类：API 调用失败（超时、5xx）和返回结果无法解析（格式错误、缺字段）。

三层处理：Prompt 层强制 JSON 格式，减少解析失败的频率；解析层 Pydantic 校验+重试，最多重试 3 次，每次等待时间指数增加；3 次全部失败后，任务状态变为 NEEDS_HUMAN，不是直接 FAILED。

NEEDS_HUMAN 的设计理念是：AI 搞不定不等于任务无法完成，人可以接管。前端会展示任务卡住在哪一步、当时的页面截图、LLM 的原始输出（用于诊断为什么失败），运维人员可以选择跳过这一步、手动执行、或者终止任务。这比直接报错对业务的影响小得多。

---

### Q6.5：为什么要做 Planner + Executor 双 Agent，而不是用原版 Skyvern 的单 Agent 循环？

原版 Skyvern 是一个感知-决策-执行的单循环 Agent，每一步都是独立的：截图→LLM 判断→执行操作→再截图。这在简单流程（登录、查余额）里没问题，但在跨多个页面、涉及多步操作的金融流程里，一旦某步失败就只能整体重来。

双 Agent 的核心价值是**失败恢复粒度**从「整个任务」细化到「单个子任务」。Planner 把"在网银系统中查询最近三个月的对公账单并导出"拆解为：登录→进入账单页面→设置查询条件→执行查询→导出结果。每个子任务独立执行、独立记录状态。第三步失败了，只需从第三步重来，前两步的结果已经持久化了。

更重要的是 replan 策略——当某条路径走不通时（比如页面结构变了，按钮不在原来的位置），不是直接失败，而是把已完成的步骤和失败信息交给 Planner 重新规划剩余步骤。这在金融系统页面频繁微调的环境里很有用。

重规划次数有上限（默认 3 次），超出后自动转 NEEDS_HUMAN。这个限制防止 Planner 和 Executor 在无解的任务上无限循环消耗 LLM 调用。

---

### Q6.6：Skill 库的设计是怎么来的，为什么不直接在工作流模板里写导航目标？

原版 Skyvern 的工作方式是给一段自然语言导航目标（如"登录到 XX 系统，然后查询 YY 数据"），让 LLM 理解并执行。这对单次灵活操作很好，但在企业场景下有三个问题：

第一是**重复浪费**。所有银行场景都要登录，每次都让 LLM 重新理解"先登录"这个步骤，浪费 token 且增加出错概率。封装成 LoginSkill 后，登录逻辑固定，只有页面结构理解的部分交给 LLM。

第二是**参数模糊性**。自然语言里写"用账号 admin 密码 123 登录"，LLM 需要从文本中提取参数。Skill 的 Pydantic 参数模型（LoginParams: url, username, password）在代码层面就约束了参数类型和必填性，消除了歧义。

第三是**维护成本**。金融系统的登录页经常变化（新增验证码、改变字段名）。LoginSkill 改一处，所有使用它的工作流都受益。如果每个工作流模板各自描述"先登录"，改一个登录逻辑要改十几个模板。

7 个 Skill 的错误策略也是差异化的：LoginSkill 默认 ABORT（登录失败后续无意义）、TableExtractSkill 默认 RETRY（表格可能因加载延迟暂时不可见）、PaginationSkill 默认 SKIP（翻页失败只影响部分数据完整性）。模板可以通过 `error_strategy_override` 覆盖默认策略。

## Part 2：系统设计

### Q7：权限这么复杂，每次查询都要做三维度过滤，性能会不会有问题？

过滤逻辑分两个层次。

请求进来时，从 JWT 的 Payload 里解析出用户的可见部门列表和可见业务线列表（这两个列表在 Token 签发时就计算好了，不是每次查询时重新计算）。然后这个信息写入 ContextVar，SQLAlchemy Session 层的全局监听器拿到这个列表，在 SELECT 时追加 `department_id IN (...)` OR `business_line_id IN (...)` 的过滤条件。

性能风险点是用户关联的部门和业务线列表很长时，IN 子句会比较大。对于普通操作员，这个列表通常只有 1-3 个值；对风控/合规用户，他们有全组织权限，会跳过这个过滤，不存在大 IN 的问题。所以实际上性能影响很小。

### Q8：如果金融机构要求审计日志留存 5 年，MinIO 的数据量怎么管理？

这是一个随时间增长的问题，解法分阶段：

索引优化是第一步，`audit_logs` 表在 `(org_id, executed_at)` 上建复合索引，查近期数据性能不会随总数据量线性下降。

数据分层是第二步，当数据量到达一定规模，按月建分区表，旧分区可以压缩或迁移到低成本存储层，查询层做透明路由。

截图的存储策略是独立考虑的——截图占空间大但查询频率低，适合按时间分桶（MinIO 按年月自动建 bucket），对于超过一定时间（比如 2 年）的截图，可以迁移到对象存储的低频访问层，查询时按需还原预签名 URL。

数据库和截图存储要分开规划，不要混在一起考虑。

## Part 3：技术原理

### Q9：Skyvern 的工作原理是什么，你读过哪些核心代码？

核心是一个「感知 → 决策 → 执行」的循环，对应三个目录：`webeye`（感知）→ `forge/agent`（决策）→ `webeye/actions`（执行）。

具体流程：Playwright 截图 → DOM 解析器提取可交互元素并打上 unique-id 标记 → 截图+DOM 信息送给多模态 LLM → LLM 返回 JSON 指定操作哪个元素 → Playwright 执行该操作 → 再截图确认结果 → 进入下一轮。

读得比较仔细的是 `forge/agent/agent.py`（主循环逻辑）和 `webeye/actions/handler.py`（action 执行器），这两个文件加起来是 Agent 最核心的部分，弄清楚这个循环是怎么运转的，才能知道在哪里插入高危操作拦截、审计日志等企业扩展逻辑。

---

### Q10：JWT 里存了 org_id 和部门列表，这个安全吗？

JWT 的安全性来自签名机制，Payload 是 Base64 编码可以被任何人读取，但无法篡改——改了签名就失效，服务端验签时会拒绝。所以把 org_id 和部门列表放在 Payload 里，安全性没有问题：攻击者可以读出来，但改不了。

真正的风险是 Token 泄漏后被他人持续使用。应对措施：合理的过期时间（本项目 8 小时），敏感操作不完全依赖 Token（高危操作还需要经过审批流），HTTPS 传输。更高安全要求的场景可以在 Redis 里维护 Token 黑名单实现强制失效，但这会增加每次请求的 Redis 查询，是个性能和安全的权衡。

---

### Q11：手写 SVG 图标而不用图标库，这个决策是怎么来的？

几个考量点：

项目里需要的图标集合是固定的，大概 20 来个，用图标库是为了用 20 个图标而引入整个库，包体大且有不用的东西。

更重要的是风格控制——毛玻璃 UI 的图标要和整体设计语言一致（stroke 粗细、端点风格、视觉重量），图标库提供的图标风格参差不齐，很难做到完全统一。手写 SVG 可以精确控制每一个参数。

维护成本上，图标数量固定的情况下，20 个 SVG 文件比 1 个图标库依赖更容易维护，改一个图标改一个文件就行，不用担心升级兼容性问题。

## Part 4：金融场景

### Q12：金融合规对 RPA 的要求具体是什么？

三个核心要求：操作留存、人工审批、数据不出网。

操作留存是监管对「可追溯性」的要求，出了问题要能查清楚「谁在什么时间让 AI 做了什么」。这对应本项目的全链路审计系统——每步操作前后截图+元数据构成完整时间线。

人工审批对应金融机构内控的「双重控制」原则，高风险操作不能一个人说了算，AI 也不行。这对应审批引擎，而且创建任务的人（operator）和审批执行的人（approver）不能是同一个人，这个约束是数据库级别强制的。

数据不出网是数据主权要求，客户的账单截图、交易记录不能传到外部云服务。这对应选择私有化 MinIO 而不是阿里云 OSS 或 AWS S3。

---

### Q13：三个行业的工作流模板，你是怎么选出这六个场景的？

选择标准是：重复性高、操作结构相对固定、不涉及实时交易执行（降低 critical 风险）、传统 RPA 容易失效的。

银行选了账单采集（对账/税务高频需求，外部网银页面会改版）和还款提醒查询（信贷经理日常批量跟进，逐一登录系统效率极低）。

保险选了理赔案件状态查询（理赔专员每天要跟进几十个案件，纯人工查询时间浪费大）和保单到期核查（客户经理主动维护的场景）。

证券选了研报归档（投研部门每天从多个券商网站下载报告，手动整理低效）和基金净值采集（资管产品监控需要定期采集净值数据）。

这六个场景的共同点是：页面结构不完全固定（有时会改版），对接的是外部系统（无法控制 UI），操作本身是只读或轻度操作（不触发 critical 高危）。Skyvern 的语义理解能力在这类场景里优势最明显。

### Q14：如果一个真实银行的技术部门要评估引入这套方案，他们会问哪些问题？

他们通常不会先问技术细节，会先问三类问题：

第一类是「出了问题怎么办」——RPA 任务失败了怎么告警？能不能快速定位问题原因？是否有降级机制？对应本项目的 NEEDS_HUMAN 状态、全链路截图、审计日志检索。

第二类是「怎么跟现有系统集成」——能不能对接我们行内的 IAM（统一身份认证）？审批能不能接入我们已有的 OA 流程？LLM 能不能用私有化部署的模型？这些是 MVP 阶段没有做但接口留着的扩展点，需要诚实说明。

第三类是「成本和合规」——LLM 调用费用是多少？数据是否留在内网？有没有通过等保评测？运营成本方面，模型路由+action 缓存的成本控制设计是对这类问题的回应；数据合规方面，MinIO 私有化部署是直接应答；等保是运营层面的事，技术架构上已经为此做了准备。

## Part 5：综合追问

### Q15：整个项目里你认为最难的决策是哪个？

（这道题建议用自己的真实感受回答，以下仅供参考思路）

最难的不是技术决策，而是「边界设定」——在哪些地方让 AI 自主执行，在哪些地方强制人工介入。

一开始的直觉是自动化程度越高越好，减少人工介入就是 RPA 的价值。但在设计金融场景的审批流时意识到，这个直觉在企业里是反的。银行的风控文化是「宁可多一道确认，不能少一道把关」，不是「AI 判断对了就行」。

所以 NEEDS_HUMAN 状态不是一个「失败兜底」，而是一个「正常状态」——对于 AI 处理不好的场景，清晰地交还控制权给人，比强行自动化处理然后出错要好得多。这个认知影响了很多具体的设计决策：审批流的存在不是「不信任 AI」，而是「给人一个可控的介入点」；NEEDS_HUMAN 不是失败，而是一种预期中的结果。

这个想法在技术上不难实现，但想清楚这件事花了不少时间。

### Q16：双 Agent 架构和 Skill 库是怎么协作的？

这是两个不同层次的抽象。

Planner Agent 负责「做什么」——接收一个高层目标（如"查询本月对公账单"），拆解为子任务序列（登录→进入账单页→设置条件→查询→导出）。每个子任务是一个语义描述，不涉及具体操作细节。

Skill 库负责「怎么做」——每个 Skill 封装了一类具体操作的执行逻辑。LoginSkill 知道怎么在页面上找到用户名密码输入框并填写，TableExtractSkill 知道怎么从 HTML 表格中提取结构化数据。

两者的连接点是工作流模板。模板中的 `skill_steps` 定义了一组 SkillStepDefinition，每个 step 指定使用哪个 Skill、参数怎么映射。Pipeline 执行器按序执行这些 Skill，每步有独立的错误策略。

Planner 规划出的子任务，如果对应到工作流模板中已有的 Skill 序列，可以直接走 Pipeline 执行路径（跳过 LLM 决策，效率更高）。如果子任务不匹配任何已有 Skill（比如临时需要处理一个从未见过的页面），则走原始的 Skyvern 单步 LLM 决策路径。两条路径共存，Skill 覆盖的场景走快路径，没覆盖的走通用路径。

这个分层设计的好处是：Skill 库的覆盖范围可以逐步扩展，每新增一个 Skill，就多一类操作不再依赖 LLM 实时决策，成本和速度都得到优化。
