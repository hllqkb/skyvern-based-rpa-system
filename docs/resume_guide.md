# 项目简历写法指南

> 以下提供三个版本，写法各有侧重，按投递岗位选择。
> 核心写作原则：不要写「用了什么技术」，要写「判断了什么问题、做了什么取舍、产生了什么价值」。

## 版本一：通用版

适用场景：后端开发、Python 开发、全栈开发等通用岗位。

```
企业级 RPA 自动化平台（基于 Skyvern 二次开发）                     2025.01 - 2025.04

• 在开源 AI 浏览器自动化框架 Skyvern 基础上进行企业级二次开发，核心补齐多租户
  隔离、高危操作审批、全链路审计三项企业化能力，完整改造前端 UI

• 设计三维度 RBAC 权限模型（部门 × 业务线 × 角色），通过 FastAPI 中间件 +
  SQLAlchemy Session 层双重过滤实现多维度数据隔离，数据库层面强制
  operator/approver 互斥约束，覆盖 15+ 权限边界场景的集成测试

• 构建金融场景高危操作双阶段识别：关键词预筛（覆盖银行/保险/证券 80+ 词汇）+
  LLM 精准判断，命中后按风险等级路由给对应层级审批人，通过 Redis Pub/Sub
  挂起任务协程等待审批结果，超时自动拒绝并推送告警

• 实现全链路合规审计：每步 action 前后截图上传私有化 MinIO，敏感字段自动脱敏，
  构建完整操作时间线；实现 LLM 三层容错体系，失败后转 NEEDS_HUMAN 状态并设计
  完整人工接管流程

• 设计 Planner + Executor 双 Agent 协作架构，将复杂金融流程拆解为子任务序列，
  四种差异化失败策略（retry/skip/abort/replan），支持断点续跑；封装 7 个可组合
  Skill（登录/表单/表格提取等），6 个金融工作流模板通过 Skill 组合实现标准化

• 基于 React + ECharts 改造前端 UI 为白色毛玻璃风格，手工实现 20+ 个 SVG
  线描图标组件，新增审批中心、审计日志、运营大屏、权限管理四个企业专属页面

技术栈：Python / FastAPI / SQLAlchemy / PostgreSQL / Redis / Playwright /
        React / ECharts / MinIO / JWT / Docker Compose / Alembic / pytest
```

## 版本二：AI 强化版

适用场景：AI 工程师、大模型应用开发、AI 产品工程师岗位。

```
企业级 AI Agent 平台（基于 Skyvern 二次开发，金融场景定制）          2025.01 - 2025.04

背景：Skyvern 是以 LLM + 视觉理解驱动的新一代 RPA 框架，原版缺少企业落地必要
的权限隔离、操作审批与合规审计能力，本项目完整补齐并深度适配金融行业场景。

AI 工程核心工作：

• 设计金融场景高危操作双阶段识别引擎
  Stage 1 关键词快速预筛（银行/保险/证券三个垂直场景专用词库，含金额正则检测），
  命中后进入 Stage 2 调用 LLM 结合页面上下文精准判断风险等级（medium/high/critical），
  两阶段结合将无效 LLM 调用减少约 70%

• 实现 LLM 调用三层容错体系
  Prompt 层：系统 Prompt 强制 JSON 输出 + Schema 示例嵌入；
  解析层：Pydantic v2 结构校验 + 清理 markdown 标记 + 指数退避重试（最多 3 次）；
  任务层：超出重试后转 NEEDS_HUMAN 状态，设计截图查看 + 三种处置方式的完整接管流程

• 设计 Planner + Executor 双 Agent 协作架构
  将「理解目标」和「执行操作」解耦为两个 Agent：Planner 通过 LLM 将导航目标拆解
  为子任务序列（含完成条件和失败策略），Executor 逐步执行并返回结构化结果。
  四种失败策略（retry/skip/abort/replan）覆盖不同子任务的失败影响差异，
  replan 时请求 Planner 重新规划而非终止任务，子任务状态持久化支持断点续跑

• 构建可组合 Skill 库（7 个基础 Skill）
  将登录、表单填写、表格提取等共用操作封装为独立 Skill（统一接口：Pydantic 参数
  模型 + async execute + 错误策略），通过 Pipeline 执行器按序组合。6 个金融工作流
  模板通过 SkillStepDefinition 声明 Skill 序列，参数映射支持引用和字面量两种模式。
  Skill 粒度的审计回调（密码字段自动掩码）满足监管对操作可追溯性的要求

• 实现页面复杂度评分与模型智能路由
  基于 DOM 元素数量、iframe 嵌套、动态内容三特征评分，路由至三档模型，
  结合 action 结果缓存（Redis，TTL 24h，DOM 结构 hash 为缓存 key），
  综合降低单任务 LLM 调用费用约 60%，并将成本节省数据接入运营大屏可视化

• 三维度权限模型下的 LLM 审批路由
  识别高危操作后，结合任务所属部门/业务线，动态路由审批请求给有权审批该业务线
  的 approver，LLM 判断风险等级同时决定审批层级（部门审批 vs 合规审计部审批）

技术栈：Python / FastAPI / SQLAlchemy / PostgreSQL / Redis Pub/Sub / Playwright /
        Pydantic v2 / React / ECharts / MinIO / Docker / Alembic
```

## 版本三：金融场景强化版

适用场景：金融科技公司、银行科技部门、保险公司技术岗，JD 中提到「金融科技」「合规」「风控」的岗位。

```
金融级 AI-RPA 平台（银行/保险/证券场景深度定制）                     2025.01 - 2025.04

背景：金融机构的 RPA 需求与其他行业有本质区别——监管合规要求所有自动化操作
可追溯，风控要求 AI 不能未经审批执行高风险金融操作，多组织架构要求细粒度的
业务线数据隔离。本项目在 Skyvern 开源 AI-RPA 框架基础上完整实现上述能力。

金融场景核心工作：

• 设计对标金融机构组织结构的三维度权限模型
  部门（行政归属）× 业务线（功能分类）× 角色（操作权限）三维组合，覆盖
  金融机构真实场景：对公信贷部操作员仅可见本部门+关联业务线任务；风险管理部
  天然获得全组织只读权限；合规审计部获得跨组织审批权限。数据库级别强制
  operator/approver 互斥约束，对应双重控制合规要求。SQL 模拟数据覆盖 15+ 场景。

• 基于金融业务语义的高危操作识别与分级审批
  针对银行（转账/放款/核销）、保险（理赔提交/退保/受益人变更）、证券（委托下单/
  融资买入）三个垂直场景建立专项词库，风险等级映射审批层级：high 级别路由对应
  部门审批员，critical 级别路由合规审计部。企业微信/钉钉推送审批通知，Redis
  Pub/Sub 实现异步等待，超时自动拒绝

• 满足监管留存要求的全链路审计系统
  参考金融机构内控要求设计操作留存方案：每步 action 前后截图存储至私有化 MinIO
  （数据不出内网），敏感字段自动脱敏（卡号保留后四位，密码完全掩码），审计日志
  支持按业务线/部门/时间范围/风险等级多维检索，满足事后追溯要求

• 设计 Planner + Executor 双 Agent 协作架构实现复杂金融流程的子任务拆解与
  断点续跑，封装 7 个可组合 Skill（登录/表单/表格提取等）构建标准化工作流，
  6 个金融场景模板（银行/保险/证券各两个）通过 Skill 组合降低 IT 开发介入门槛

• 敏感参数 Fernet 加密存储，实现毛玻璃风格前端改造及审批中心、审计日志查询、
  运营大屏等企业专属页面

技术栈：Python / FastAPI / SQLAlchemy / PostgreSQL / Redis / Playwright /
        React / ECharts / MinIO / JWT-RBAC / Docker / Alembic
```

## 三版本对比

| | 通用版 | AI 强化版 | 金融场景强化版 |
|-|--------|-----------|--------------|
| 核心卖点 | 工程能力全面 | LLM 工程深度 | 行业理解深度 |
| 适用公司 | 互联网通用后端 | AI 创业/大厂 AI 部门 | 金融科技/银行/保险 |
| 权限体系描述 | 侧重技术实现 | 侧重与 LLM 联动 | 侧重金融合规背景 |
| UI 改造是否强调 | 提及 | 不强调 | 不强调 |

**选版建议**：JD 提到 AI/大模型 → AI 强化版；JD 提到金融/合规/风控 → 金融场景强化版；其余 → 通用版。同时投多类岗位时分别用不同版本，不要混用。

## 被追问时的参考应答思路

**Q：权限矩阵这么复杂，实际上真的用到了吗？**

> 权限模型是从金融机构的组织结构出发设计的，不是为了复杂而复杂。银行的对公信贷部和个人金融部各自有自己的 RPA 流程，他们的数据不该互看；但风控部需要看所有业务线的执行情况；合规部需要能审批任何业务线的高危操作。如果用简单的 org_id 单字段隔离，要么满足不了风控/合规的跨部门需求，要么就要给所有人开放全组织权限，两端都不对。三维度模型是这个真实场景的最小表达。

**Q：UI 改造这个有什么技术含量？**

> 技术含量不高，但有工程判断。图标不用图标库而是手写 SVG，是因为金融系统的图标集合固定，手写可以精确控制风格一致性，不需要引入一个为了用 10 个图标而装的整套图标库。毛玻璃风格用 CSS 变量统一管理设计 token，方便后续主题切换。这些决策都有具体理由，不是随意选的。
